\section{Methods}\label{sec:Methods}
In this section we describe the methods used to model biodiversity at the community level. We first outline our major contribution---a pipeline for spatial biodiversity modeling---and then describe the data, model selection process, and evaluation metrics used in our study.
\subsection{Spatial Biodiversity Modeling Pipeline}\label{subsec:pipeline}
% Rename section to something with spatial biodiversity pipeline
% Was built on Jakob's pipeline and cite. 
The spatial biodiversity modeling pipeline we developed is based on the work of Nyström and the infrastructure pipeline developed for that Master's thesis \parencite{nystrom2024}. The SBM pipeline consists of the following three main components: data processing, feature engineering, and model training. The goal of the SBM pipeline is to facilitate the development and evaluation of top-down community-level biodiversity models and to provide a tool for ecologists and researchers to explore the potential of these models in predicting biodiversity patterns and trends.  With this in mind, we designed the SBM-pipe to be flexible in terms of the types of data and models that can be used, allowing for this wide range of use cases. The SBM pipeline is implemented in Python and will be released to the public at a later date \parencite{sbm-pipe}. All configurations mentioned throughout this section are specified in a configuration file, allowing for easy customization. A diagram of the pipeline is shown in Figure \ref{fig:pipeline_diagram}, and a more detailed description of each step is provided in the following subsections.

\subsubsection{Biodiversity data input}
The first step in the SBM pipeline is to select appropriate biodiversity data. Researchers can draw on a wide range of biodiversity data sources, including citizen-science initiatives, government-funded monitoring programmes, and researcher-collected datasets \parencite{zipkin2023}; however, to standardize the data input we chose to use the Darwin Core (DwC) format \parencite{wieczorek_darwin_2012}. This format is a flexible and straightforward format for biodiversity data. Additionally, one of the largest biodiversity data repositories, GBIF, standardizes its data according to this format. Therefore, the SBM works with DwC data formats as this accommodates all GBIF biodiversity datasets natively. While the DwC format is widely used and has good standardization, it is important to note that there are many other formats for biodiversity data, and the SBM pipeline is designed to account for this by allowing other formats as long as the user provides a mapping to DwC format. Currently, the SBM pipeline accepts data in a \texttt{.csv} or \texttt{.txt} file format, but in the future we plan to extend this to use the GBIF API interface as well \footnote{Global Biodiversity Information Facility (GBIF). \textit{GBIF API Reference}. Available at: \url{https://techdocs.gbif.org/en/openapi/} (Accessed: 17 May 2026).}. 

\subsubsection{Data Processing}\label{subsubsec:data_processing}
Once the biodiversity dataset is selected by the user, the next step in the SBM pipeline is to process the data. This involves several steps, including ingesting and cleaning the data, site polygon creation, and environmental feature extraction. 

\begin{enumerate}
\item \textbf{Ingestion and data cleaning.}

After the dataset is loaded, the SBM pipeline removes any records with missing or invalid data, removes rows that don't have precise location information, removes records that are outside the dataset's date range, removes exact duplicates, validates location identification integrity, and infers organism quantity \footnote{GBIF data quality guidance: \url{https://docs.gbif.org/course-data-mobilization/en/data-quality.html} (accessed 2026-05-12).}. Important steps here involve removing rows that are not precise enough based on the latitude and longitude precision. Validating location identification integrity is important because if a biodiversity dataset is collected over many decades, it is possible that the same unique location identifier (location ID) may change for latitude and longitude pairs. This results in many location IDs mapping to the same latitude and longitude values, which can cause issues. To address this, the pipeline checks for duplicate location IDs based on latitude and longitude values and will reassign location IDs, if applicable, to ensure that location ID is a 1:1 mapping to latitude and longitude pairs. Finally, the pipeline can infer organism abundance by checking for the presence of the "individualCount", "organismQuantity", or "organismQuantityType" fields in the dataset\footnote{Biodiversity Information Standards (TDWG). \textit{Darwin Core Terms}. Available at: \url{https://dwc.tdwg.org/terms/} (Accessed: 17 May 2026).}\(^,\)\footnote{Biodiversity Information Standards (TDWG). \textit{Darwin Core Terms}. Available at: \url{https://rs.tdwg.org/dwc/terms.htm} (Accessed: 17 May 2026).}. If these fields are not present or null then the pipeline will infer organism abundance by grouping the observations based on location ID, species, and date. This is done because sometimes abundance data is separated into the individual observations. If that data is truly presence-only or presence-absence then this will have no effect on the dataset. This step will output \texttt{.parquet} files containing the cleaned data and site locations.

\item \textbf{Site polygon creation}

Site polygon creation involves creating polygons around each unique latitude and longitude pair in the dataset at different scales. This is done because biodiversity datasets usually vary in spatial scale \parencite{sicacha-parada2022} and we want to capture the surrounding environment across a range of spatial scales. The polygons are also created so we can later extract environmental features from them. The user can define the radius for the site polygons as well as the shape of these polygons. If the dataset contains coordinate uncertainty information, then the pipeline will also create polygons with a radius equal to the coordinate uncertainty value for each site. These polygons are projected both in a global coordinate reference system (CRS) and a local CRS, which is determined by the latitude and longitude values of each site. The global CRS, EPSG:4326, is used for extracting environmental features, while the local CRS in UTM format can be used later for visualizations. All these polygons are saved as \texttt{.shp} files for later use in the pipeline.

\item \textbf{Environmental feature extraction}
Once every site polygon is created, the next step is to extract environmental features from these polygons. The environmental features are extracted from datasets provided by the user as the choice of environmental features can vary and is not the same for every problem \parencite{williams2012}. The environmental feature datasets should be provided in a raster format; this will allow the SBM pipeline to extract environmental features from the site polygons of varying sizes. The user can also specify the type of summary statistic to use when extracting features from the site polygons; this includes the mean, median, minimum, maximum, and standard deviation of the feature. Once this step is complete, the SBM pipeline output \texttt{.parquet} files containing the extracted environmental features for each site at the spatial scale of the site polygon.
\end{enumerate}

\subsubsection{Feature Engineering}\label{subsubsec:feature_engineering}

Once the data has been processed, site polygons created, and environmental features extracted, the next step in the SBM pipeline is to perform feature engineering. This involves several steps, including combining datasets, creating new features from the existing data, and calculating community-level biodiversity metrics.

\begin{enumerate}
\item \textbf{Combining datasets}

The first step in feature engineering is to combine the biodiversity dataset from step \(1\) with the environmental features from step \(3\) in Section~\ref{subsubsec:data_processing}. This involves joining the datasets based on the location ID for each site and time dimension which is usually the year. This results in a dataset that contains both the biodiversity data and the environmental features for each site and time dimension. Furthermore, if population density features are included in the environmental features, the pipeline will also interpolate the population density between sampled years if applicable. This is done because population density data is might only be provided for certain years, and we want to have population density values for each year in our dataset. 

The interpolation is done based on exponential growth or decay and depends on the population density values in the provided years. The population density in non-sampled years is calculated by \(P_t = P_s * e^{r(t-s)}\), where \(P_t\) is the population density at time \(t\), \(P_s\) is the population density at the nearest year \(s\) before time \(t\), \(r\) is the growth rate. The growth rate is determined as \(\frac{P_e - P_s}{e-s}\), where \(P_e\) is the population density at the nearest year \(e\) after time \(t\). We can then interpolate the population densities between provided years and extrapolate population densities for years before the earliest provided year and after the latest provided year by calculating the growth rate between the two nearest provided years. 

\item \textbf{Feature engineering}

Once we have our unified dataset, the next step is feature engineering. Feature engineering involves creating new features from the existing data that may be more informative for modeling. This can include creating interaction terms between environmental features, creating polynomial features, or other advanced feature engineering techniques \parencite{Katya2023}. Currently, the feature engineering in the SBM pipeline will log, square, and square root transform the continuous environmental features. Categorical features are one-hot encoded. The user can specify which transformations to apply to which features, and the SBM pipeline will output a \texttt{.parquet} file containing the engineered features in addition to the original features for each site and time dimension.

\item \textbf{Calculating community-level biodiversity metrics}

An important step of the feature engineering process is to calculate the community-level biodiversity metrics that will be used as the target variable for modeling. Currently, the SBM pipeline calculates the alpha diversity metrics described in Section~\ref{subsec:biodiversity} except for relative abundance and calculates a modified Shannon index value which is \(H_{modified} = \ln(A_{r_{i}} + 1) * H\). Furthermore, when calculating these alpha diversity metrics, the user can define the taxonomic resolutions to group at before calculating these metrics. By default no taxonomic grouping done. This type of taxonomic grouping is sometimes used when species-level identification is difficult, costly, or inconsistent \parencite{de_oliveira_higher_2020}. At the end of this step, the SBM pipeline outputs a \texttt{.parquet} file containing the community-level biodiversity metrics for each site and time dimension along with the environmental features. This final dataset can then be used for model training and evaluation.
\end{enumerate}


\subsubsection{Model Training}\label{subsubsec:model_training}

After we have performed feature engineering and computed our community-level biodiversity metrics, the next step in the SBM pipeline is to split the data for training and testing and train a model. This involves the following steps: determining the type of training to perform, selecting or developing a model, and training and evaluating the model.

\begin{enumerate}
\item \textbf{Model dataset preparation and splitting}\label{subsubsec:model_dataset_preparation}

This step begins with determining the final filtering and the type of training to perform. This final filtering can be done based on criteria such as removing outliers, filtering based on biogeographic factors, or taxonomic scope. This can be done to further customize the model training process and not repeat the more time consuming data loading and processing steps. Here the user also specifies the community-level biodiversity metric to estimate, such as species richness or abundance.

The SBM pipeline is designed with three different training types: training on the entire dataset, training on a subset of the data and having a hold-out set (train-test), or cross-validation. The train-test and cross-validation training modes also have additional settings to determine the type of split used. As described in Section~\ref{subsec:current-approaches}, biodiversity data is frequently spatially autocorrelated, and therefore special care must be taken when splitting the data for training and testing to account for this \parencite{elith2009,wang2023}. Therefore, the SBM pipeline implements both random and spatial cross-validation strategies. The user can specify which type of training to perform and the type of split to use. Spatial cross-validation is implemented using the K-nearest neighbors algorithm \parencite{scikit-learn} to generate the spatial blocks based on a user defined criteria in the dataset. This separates the data spatially and allows the user to test the model's performance on a variety of different data splits. This further promotes the evaluation of a model's generalization performance in a robust way. A visualization of the difference between random cross-validation and spatial cross-validation can be seen in Figure \ref{fig:cv_diagram}.

\item \textbf{Model Development}

Once the type of training is determined, along with a splitting strategy if needed, a top-down model is chosen or developed. Top-down modeling approaches are a broad class from regression techniques, statistical techniques, and machine learning techniques \parencite{ferrier2002, olaya-marin_comparison_2013, mondal2021, andermann2022, brugere2023}. We have designed the SBM pipeline to accommodate this wide range of modeling techniques, allowing researchers to explore different approaches and compare their performance. This integration is done through a superclass called \texttt{Model}. This class defines the behavior for every model that is trained in the SBM pipeline, and allows for easy integration of new models. A new model only needs to inherit from the \texttt{Model} class and implement the \texttt{train\_model} and \texttt{get\_predictions} methods. Currently, the SBM pipeline has generalized linear models (GLMs), generalized additive models (GAMs), random forests, CatBoost, and FFNNs implemented, but the pipeline is designed to allow for easy integration of custom models in the future. The user can choose one of the current model implementations with specific hyperparameters defined in the \texttt{params} block of each model's configuration file, or they can build their own custom model and specify that in the configuration file. The flexibility of the SBM pipeline allows for a wide range of modeling techniques to be used and promotes the exploration of different approaches to modeling biodiversity patterns and trends at the community level.

\item \textbf{Model training and evaluation}\label{subsubsec:model_training_and_evaluation}

After the type of training and model has been determined, the model is trained on the dataset. This is done by specifying the model type and the location of a configuration file. Scaling or transforming the predictor variables can also be configured at this step. The model is then trained to predict the community-level biodiversity metric specified by the user, and several metrics will be output. These metrics include: predictions on both the training and testing datasets, if applicable, \(R^2\) values, mean absolute error (MAE), mean absolute percentage error (MAPE), symmetric mean absolute percentage error (SMAPE), root mean squared error (RMSE), Pearson correlation coefficient, and Spearman correlation coefficient. Additionally, the bias ratios for the train and test datasets are included, along with the bias ratios for the top and bottom quartiles of the train and test datasets. These metrics summarize predictive accuracy, bias, and rank agreement, providing insight into model behavior and suitability for predicting community-level biodiversity patterns. More specific information on how these metrics are calculated will be described in Section~\ref{subsec:model_comparison}.
\end{enumerate}

\subsubsection{Model Application}\label{subsubsec:model_application}
The outputs from the SBM pipeline can then be used in downstream applications. This can be anything from decision making \parencite{IPBES2019spm} to mapping biodiversity patterns, which is commonly done in the literature \parencite{andermann2022, baggstrom2025}. The goal is that the SBM pipeline can be used as a tool for researchers to explore the potential of top-down community-level biodiversity models and to expand this research field. The specific application is not defined in the SBM pipeline as this can vary widely depending on the research question and context and is rather left to the user to decide.

\subsection{Application of the SBM Pipeline}\label{subsec:application}
In this subsection we apply the SBM pipeline to a case study of biodiversity data from the United Kingdom Butterfly Monitoring Scheme (UKBMS) acquired from GBIF \parencite{UKBMS2026}. We process these data with the SBM pipeline and then develop five different top-down models to predict species richness. These models are then compared against each other, across different cross-validation strategies, and in data-limited scenarios. The details of the data used in this case study are described in Section~\ref{subsubsec:data}; the model selection process is described in Section~\ref{subsubsec:models_spefication} along with the evaluation metrics that will be used to compare the models in Section~\ref{subsec:model_comparison}. The experiments of this case study are described in Section~\ref{sec:Experiments} and the discussion of the results are discussed in Section~\ref{sec:discussion}.

\subsubsection{Data}\label{subsubsec:data}

To demonstrate the practical application of the SBM pipeline we process the United Kingdom Butterfly Monitoring Scheme (UKBMS) dataset with the pipeline. This dataset is a long-term monitoring program that collects abundance records of butterflies across the United Kingdom and Ireland \parencite{UKBMS2026}. The temporal scale of this dataset is from May 1, 1973 - October 27, 2020, however; when loading the data the date range extends to October 27, 2024. The data is collected from defined transect walks every week from the beginning of April to the end of September, weather permitting, ideally providing 26 sample events per site per year. A transect walk is conducted by individuals walking along a defined transect, or route, and recording butterflies they observe, along with the species and abundance of these butterflies \parencite{pollard1993}. At the time of writing this dataset has \(10,209,486\) occurrence records. 

We processed this dataset through the SBM first for ingestion and cleaning as described in step \(1\) of Section~\ref{subsubsec:data_processing}. An important note is that butterfly counts were inferred because abundance records were separated into individual observations; therefore, records were grouped by latitude, longitude, and event date. Our coordinate precision was checked down to four decimal places, which is roughly 10-meter precision or similar \parencite[Table~1]{daniel2016}. Standard filters of invalid data and invalid dates were also automatically applied. After ingestion and cleaning there were \(3,673,775\) abundance records across \(3,038\) unique sites. The spatial distribution of all the sites in the UKBMS dataset can be seen in Figure \ref{fig:all_sites_distribution}.

After data ingestion and cleaning, site polygons were constructed and environmental predictor variables were extracted in line with steps \(2\) and \(3\) of Section~\ref{subsubsec:data_processing}. We created polygons with radii \(1,5,10,50\) km in addition to the coordinate uncertainty of the sites, which was \(707.1\) m. We then extracted environmental features from each site from three sources: climatic variables from WorldClim \parencite{fick2017worldclim}, population density from the Gridded Population of the World version 4 (GPWv4) dataset \parencite{ciesin2017gpwv4}, and topographic variables from EarthEnv \parencite{amatulli2018topographic}. The extracted variables and spatial scales of the radii used are summarized in Table~\ref{tab:environmental-features}. The climatic features we extracted include: annual mean temperature, temperature seasonality, maximum temperature of the warmest month, minimum temperature of the coldest month, annual precipitation, precipitation of the wettest month, precipitation of the driest month, and precipitation seasonality. The topographic features we extracted include: elevation, slope, terrain roughness, terrain ruggedness index, aspect eastness, and aspect northness. The population density feature we extracted were the population densities across the years 2000, 2005, 2010, 2015, and 2020. All datasets had data available at a \(0.5^\circ\) degree resolution or approximately a \(1\) km spatial resolution. Environmental feature values were extracted by averaging all raster cells overlapping each site polygon. Differing sizes were used to capture the surrounding environmental context at different neighborhood sizes, as biodiversity and species patterns can be influenced by environmental factors at varying spatial scales \parencite{amatulli2018topographic, mertes2018scale}.

\subsubsection{Model Specification}\label{subsubsec:models_spefication}

For the demonstration of the SBM pipeline we train a GLM, GAM, random forest, CatBoost, and a FFNN for species richness prediction. These models were chosen to represent a range of different modeling approaches, including statistical models (GLM and GAM) and machine learning models (random forest, CatBoost, and FFNN) that are commonly used in biodiversity modeling \parencite{olaya-marin_comparison_2013, mondal2021, andermann2022, brugere2023, cai2023, ngor2023, baggstrom2025}. The GLM and GAM were implemented with the \texttt{glm} and \texttt{glmgam} modules, respectively, from the statsmodels library \parencite{seabold2010statsmodels}. Random forest was implemented with the \texttt{RandomForestRegressor} class from the scikit-learn library \parencite{scikit-learn}. CatBoost was implemented with the \texttt{CatBoostRegressor} class from the CatBoost library \parencite{catboostdocs2026}. We acknowledge that CatBoost is designed to natively work with categorical features; however, all of our environmental predictors are continuous, and this should not affect how CatBoost is trained. The FFNN was implemented with the \texttt{nn.Module} from PyTorch \parencite{pytorchdocs2026}. 

We ran 10-fold random cross-validation across several hyper-parameter combinations for each model to evaluate performance and to select one model per class to use for the research questions. For each hyper-parameter combination, we calculated the mean \(R^2\), median \(R^2\) and interquartile range of \(R^2\) across test folds. Models were ranked separately for each of these three metrics, where higher \(R^2\) values are better for mean and median, and lower is better for the interquartile range. The best model per class was selected based on the best cumulative rank across these three metrics. These metrics were chosen to give weight to models that on average would perform well while also being robust to individual variations in data. Reasonable values for the hyper-parameter grid search were selected, and the full hyper-parameter search space for each model class is shown in Table~\ref{tab:hyperparameter-grid-searches} in Appendix~\ref{appendix:appendix_b}. The best-performing model from each model class, selected using this evaluation, is shown in Table \ref{tab:best-hyperparameters} below.

We chose to use random cross-validation for model selection for our experiments as it is a good indicator of model performance and is commonly used \parencite{efron_estimating_1983}. We acknowledge that random cross-validation can overestimate model performance when spatial autocorrelation is present in the data \parencite{wang2023}. However, random cross-validation is still a useful tool for model evaluation and is suitable for fairly spread out observations \parencite{de_bruin_dealing_2022}. The best-performing model from each model class was then used in three comparative experiments. First, these models were compared against each other to address Research Question 1. Then, the performance of these models was compared across different cross-validation strategies to address Research Question 2. Finally, the performance of these models was compared across different data scenarios to address Research Question 3. How the models are compared against each other is described in Section~\ref{subsec:model_comparison}.

\subsection{Model comparison methods}\label{subsec:model_comparison}

The SBM pipeline outputs several evaluation metrics for each model trained,
which are described in Section~\ref{subsubsec:model_training_and_evaluation}. For our model comparison we evaluate models using \(R^2\), root mean squared error (RMSE), mean absolute percentage error (MAPE), and symmetric mean absolute percentage error (SMAPE) along with visual inspections. Each experiment in Section~\ref{sec:Experiments} is conducted with a 10-fold cross-validation strategy, and for our model evaluations we calculate these metrics across the test sets of all folds and additionally summarize these metrics across folds.

\(R^2\) or the coefficient of determination, is a commonly used metric for evaluating the performance of predictions compared to observed values. It quantifies the proportion of variance in the response variable that is predictable from the predictor variables \parencite{chicco2021}. The \(R^2\) score is calculated according to the implementation of the scikit-learn library \parencite{scikit-learn}, which is

\[R^2(y,\hat{y}) = 1 - \frac{\sum_{i=1}^n (y_i - \hat{y}_i)^2}{\sum_{i=1}^n (y_i - \bar{y})^2},\]

where \(y\) is the vector of observed values, \(\hat{y}\) is the vector of predicted values, and \(\bar{y}\) is the mean of the observed values. This \(R^2\) score can range from \(-\infty\) to \(1\), where a score of \(1\) indicates perfect predictions, a score of \(0\) indicates that the model performs no better than predicting the mean of the observed values, and negative values indicate that the model performs worse than predicting the mean. \(R^2\) values can also generally be interpreted on their own rather than scale-dependent error metrics; for example, \(R^2=0.8\) generally indicates that the model explains a large proportion of the variation in the response variable \parencite{chicco2021}. \(R^2\) scores are also common evaluation metrics for top-down models and widely understood \parencite{olaya-marin_comparison_2013, mcnellie2021, cai2023, brugere2023, baggstrom2025}.

Root mean squared error (RMSE) is another commonly used metric for evaluating model performance. It measures the distance between predicted and observed values, giving more weight to larger errors \parencite{chicco2021}. RMSE is calculated as

\[RMSE(y,\hat{y}) = \sqrt{\frac{1}{n}\sum_{i=1}^n (y_i - \hat{y}_i)^2},\]

and is implemented according to the scikit-learn library in the SBM pipeline \parencite{scikit-learn}. RMSE is useful for understanding the average magnitude of the errors in the same units as the response variable, and values can range from \(0\) to \(\infty\). RMSE is also commonly used in top-down biodiversity modeling and widely understood \parencite{mondal2021, cai2023, baggstrom2025}.

Mean absolute percentage error (MAPE) is a metric that measures the average absolute percentage difference between predicted and observed values \parencite{chicco2021}. It is calculated as

\[MAPE(y,\hat{y}) = \frac{1}{n}\sum_{i=1}^n \left|\frac{y_i - \hat{y}_i}{y_i}\right|,\]

and is implemented according to the scikit-learn library in the SBM pipeline \parencite{scikit-learn}. MAPE is useful when relative prediction error is of interest. For example, predicting a species richness of 10 when the observed value is 20 represents a larger relative error than predicting 100 when the observed value is 110, even though both predictions have an absolute error of 10. MAPE therefore gives greater weight to the first prediction because the error is larger relative to the observed value, but this measurement can be biased towards low forecasts \parencite{chicco2021}. The values of MAPE can range from \(0\) to \(\infty\). Although MAPE appears in some top-down biodiversity modeling studies \parencite{andermann2022}, it is useful for understanding relative prediction error; therefore, we include it in our model comparisons.

The final metric we based evaluations on is the symmetric mean absolute percentage error (SMAPE). SMAPE measures the absolute difference between predicted and observed values relative to half the sum of their absolute values. It is calculated as

\[SMAPE = \frac{1}{n}\sum_{i=1}^n \frac{\left|y_i - \hat{y}_i\right|}{(\left|y_i\right| + \left|\hat{y}_i\right|)/2},\]

and was designed to address some of the issues with MAPE.
SMAPE is bounded by the values of \(0\) and \(2\), where a value of \(0\) indicates perfect predictions and a value of \(2\) indicates the worst possible predictions. SMAPE is beneficial because it is bounded and can be interpreted independently rather than being based on scale dependent error metrics \parencite{chicco2021}. However, SMAPE can still penalize underestimates and overestimates differently for a fixed observed value. SMAPE is not commonly used in top-down biodiversity modeling, but it can be a useful metric for understanding the relative error of predictions while also being bounded, and therefore we include it in our model comparisons.

For these numeric metrics, higher values for \(R^2\) indicate better predictive performance, whereas for RMSE, MAPE, and SMAPE, lower values indicate better predictive performance. These four metrics were chosen because \(R^2\) gives us a measure of the proportion of variance in the response variable that is explained by the model, RMSE gives us an error in terms of the response variable, MAPE provides an unbounded measure of relative error, and SMAPE provides a bounded measure of relative error. In addition to these metrics, we also evaluate the models with visual checks. Histograms are used to assess whether the predicted values reproduce the distribution and range of the observed values. Scatter plots of observed versus predicted values are used to identify systematic bias, regression toward the mean, and poor performance at high or low richness values. We evaluate models using these metrics and visual diagnostics together, with emphasis on models that have high \(R^2\) values, low error metric values, and distributions similar to the observed values. This evaluation approach, combining numerical and visual checks, allows us to compare the performance of different models in a comprehensive way. In Section~\ref{sec:Experiments} we apply this evaluation approach to compare the performance of the different models against each other, across different cross-validation strategies, and in data-limited scenarios.
