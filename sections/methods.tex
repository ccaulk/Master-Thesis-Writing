\section{Methods}\label{sec:Methods}
In this section we describe the methods used to model biodiversity at the community level. We first outline our major contribution---a pipeline for spatial biodiversity modeling---and then describe the data, model selection process, and evaluation metrics used in our study.
\subsection{Spatial Biodiversity Modeling Pipeline}\label{subsec:pipeline}
% Rename section to something with spatial biodiversity pipeline
% Was built on Jakob's pipeline and cite. 
The spatial biodiversity modeling pipeline, we developed is based on the work of Nyström and the infrastructure pipeline developed for that Master's thesis \parencite{nystrom2024}. The SBM pipeline consists of the following three main components: data processing, feature engineering, and model training. The goal of the SBM pipeline is to facilitate the development and evaluation of top-down community-level biodiversity models, and to provide a tool for ecologists and researchers to explore the potential of these models in predicting biodiversity patterns and trends.  With this in mind, we designed the SBM-pipe to be flexible in terms of the types of data and models that can be used, allowing for this wide range of use cases. The SBM pipeline is implemented in Python and will be released for the public at a later date \parencite{sbm-pipe}. All configurations mentioned throughout this section are specified in a configuration file, allowing for easy customization. A diagram of the pipeline is shown in Figure \ref{fig:pipeline_diagram}, and a more detailed description of each step is provided in the following subsections.

\subsubsection{Biodiversity data input}
The first step in the SBM pipeline is to select appropriate biodiversity data. Researchers can draw on a wide range of biodiversity data sources, including citizen-science initiatives, government-funded monitoring programmes, and researcher-collected datasets \parencite{zipkin2023}; however, to standardize the data input we chose to use the Darwin Core (DwC) format \parencite{wieczorek_darwin_2012}. This format is a flexible and straightforward format for biodiversity data. Additionally, one of the largest biodiversity data repositories, GBIF, standardizes its data according to this format. Therefore, the SBM works with DwC data formats as this accommodates all GBIF biodiversity datasets natively. While the DwC format is widely used and has good standardization, it is important to note that there are many other formats for biodiversity data, and the SBM pipeline is designed to account for this by allowing other formats as long as the user provides a mapping to DwC format. Currently, the SBM pipeline accepts data in a \texttt{.csv} or \texttt{.txt} file format, but in the future we plan to extend this to use the GBIF API interface as well \footnote{Global Biodiversity Information Facility (GBIF). \textit{GBIF API Reference}. Available at: \url{https://techdocs.gbif.org/en/openapi/} (Accessed: 17 May 2026).}. 

\subsubsection{Data Processing}\label{subsubsec:data_processing}
Once the biodiversity dataset is selected by the user, the next step in the SBM pipeline is to process the data. This involves several steps, including ingesting and cleaning the data, site polygon creation, and environmental feature extraction. 

\begin{enumerate}
\item \textbf{Ingestion and data cleaning.}

After the dataset is loaded, the SBM pipeline removes any records with missing or invalid data, removes rows that don't have precise location information, removes records that are outside the dataset's date range, removes exact duplicates, validates location identification integrity, and inferres organism quantity \footnote{GBIF data quality guidance: \url{https://docs.gbif.org/course-data-mobilization/en/data-quality.html} (accessed 2026-05-12).}. Important steps here involve removing rows that are not precise enough based on the latitude and longitude precision. Validating location identification integrity because if a biodiversity dataset is conduct over many decades, it is possible that the same unique location identifier (location ID) may change for latitude and longitude pairs. This results in many location IDs mapping to the same latitude and longitude values, which can cause issues. To address this, the pipeline checks for duplicate location IDs based on latitude and longitude values and will reassign location IDs, if applicable, to ensure that location ID is a 1:1 mapping to latitude and longitude pairs. Finally, the pipeline can infer organism abundance by checking for the presence of the "individualCount", "organismQuantity", or "organismQuantityType" fields in the dataset\footnote{Biodiversity Information Standards (TDWG). \textit{Darwin Core Terms}. Available at: \url{https://dwc.tdwg.org/terms/} (Accessed: 17 May 2026).}\(^,\)\footnote{Biodiversity Information Standards (TDWG). \textit{Darwin Core Terms}. Available at: \url{https://rs.tdwg.org/dwc/terms.htm} (Accessed: 17 May 2026).}. If these fields are not present or null then the pipeline will infer organism abundance by grouping the observations based on location ID, species, and date. This is done because sometimes abundance data is separated into the individual observations. If that data is truly presence-only or presence-absence then this will have no effect on the dataset. This step will output \texttt{.parquet} files containing the cleaned data and site locations.

\item \textbf{Site polygon creation}

Site polygon creation involves creating polygons around each unique latitude and longitude pair in the dataset at different scales. This is done because biodiversity datasets usually vary in spatial scale \parencite{sicacha-parada2022} and we want to capture the surrounding environment across a range of spatial scales. The polygons are also created so we can later extract environmental features from them. The user can define the radius for the site polygons as well as the shape of these polygons. If the dataset contains coordinate uncertainty information, then the pipeline will also create polygons with a radius equal to the coordinate uncertainty value for each site. These polygons are projected both in a global coordinate reference system (CRS) and a local CRS, which is determined by the latitude and longitude values of each site. The global CRS, EPSG:4326, is used for extracting environmental features, while the local CRS in UTM format can be used later for visualizations. All these polygons are saved as \texttt{.shp} files for later use in the pipeline.

\item \textbf{Environmental feature extraction}
Once every site polygon is created, the next step is to extract environmental features from these polygons. The environmental features are extracted from datasets provided by the user as the choice of environmental features can vary and is not the same for every problem \parencite{williams2012}. The environmental feature datasets should be provided in a raster format, this will allow the SBM pipeline to extract environmental features from the site polygons of varying sizes. The user can also specify the type of summary statistic to use when extracting features from the site polygons, this includes mean, median, min, max, and standard deviation of the feature. Once this step is complete the SBM pipeline outputs a \texttt{.parquet} files containing the extracted environmental features for each site at the spatial scale of the site polygon.
\end{enumerate}

\subsubsection{Feature Engineering}\label{subsubsec:feature_engineering}

Once the data has been processed, site polygons created, and environmental features extracted, the next step in the SBM pipeline is to perform feature engineering. 

\begin{enumerate}
\item \textbf{Combining datasets}

The first step in feature engineering is to combine the biodiversity dataset from step \(1\) with the environmental features from step \(3\) in Section~\ref{subsubsec:data_processing}. This involves joining the datasets based on the location ID for each site and time dimension which is usually the year. This results in a dataset that contains both the biodiversity data and the environmental features for each site and time dimension. Furthermore, if population density features are included in the environmental features, the pipeline will also interpolate the population density between sampled years if applicable. This is done because population density data is usually only provided for certain years, and we want to have population density values for each year in our dataset. The interpolation is done based on exponential growth or decay and depends on the population density values in the provided years.

\item \textbf{Feature engineering}

Once we have our unified dataset, the next step is feature engineering. Feature engineering involves creating new features from the existing data that may be more informative for modeling. This can include creating interaction terms between environmental features, creating polynomial features, or other advanced feature engineering techniques \parencite{Katya2023}. Currently, the feature engineering in the SBM pipeline will log, square, and square root transform the continuous environmental features. Categorical features are one-hot encoded. The user can specify which transformations to apply to which features, and the SBM pipeline will output a \texttt{.parquet} file containing the engineered features in addition to the original features for each site and time dimension.

\item \textbf{Calculating community-level biodiversity metrics}

An important step of the feature engineering process is to calculate the community-level biodiversity metrics that will be used as the target variable for modeling. Currently, the SBM pipeline calculates the alpha diversity metrics described in Section~\ref{subsec:biodiversity} except for relative abundance and calculates a modified Shannon index value which is \(H_{modified} = \ln(A_{r_{i}} + 1) * H\). Furthermore, when calculating these alpha diversity metrics, the user can define the taxonomic resolutions to group at before calculating these metrics. By default no taxonomic grouping done. This type of taxonomic grouping is sometimes used when species-level identification is difficult, costly, or inconsistent \parencite{de_oliveira_higher_2020}. At the end of this step, the SBM pipeline outputs a \texttt{.parquet} file containing the community-level biodiversity metrics for each site and time dimension along with the environmental features. This final dataset can then be used for model training and evaluation.
\end{enumerate}


\subsubsection{Model Training}\label{subsubsec:model_training}

After we have performed feature engineering and computed our community-level biodiversity metrics, the next step in the SBM pipeline is to split the data for training and testing and train a model.

\begin{enumerate}
\item \textbf{Model dataset preparation and splitting}

This step begins with determining final filtering and the type of training to perform. This final filtering can be done based on criteria such as removing outliers, filtering based on biogeographic factors, or taxonomic scope. This can be done to further customize the model training process and not repeat the more time consuming data loading and processing steps. Here the user also specifies the community-level biodiversity metric to estimate such as species richness or abundance.

The SBM pipeline is designed with three different training types: training on the entire dataset, training on a subset of the data and having a hold-out set (train-test), or cross-validation. The train-test and cross-validation training modes also have additional settings to determine the type of split used. As described in Section~\ref{subsec:current-approaches} biodiversity data is frequently spatially autocorrelated, and therefore special care must be taken when splitting the data for training and testing to account for this \parencite{elith2009,wang2023}. Therefore, the SBM pipeline implements both random and spatial cross-validation strategies. The user can specify which type of training to perform and the type of split to use. Spatial cross-validation is implemented using the K-nearest neighbors algorithm \parencite{scikit-learn} to generate the spatial blocks based on a user defined criteria in the dataset. This separates the data spatially, and allows the user to test the model's performance on a variety of different data splits. This further promotes the evaluation of a model's generalization performance in a robust way. A visualization of the difference between random cross-validation and spatial cross-validation can be seen in Figure \ref{fig:cv_diagram}.

\item \textbf{Model Development}

Once the type of training is determined along with a splitting strategy if needed a top-down model is chosen or developed. Top-down modeling approaches are broad class from regression techniques, statistical techniques, and machine learning techniques \parencite{ferrier2002, olaya-marin_comparison_2013, mondal2021, andermann2022, brugere2023}. We have designed the SBM pipeline to accommodate this wide range of modeling techniques, allowing researchers to explore different approaches and compare their performance. This integration is done through a super class called \texttt{Model}. This class defines the behavior for every model that is trained in the SBM pipeline, and allows for easy integration of new models. A new model only needs to inherit from the \texttt{Model} class and implement the \texttt{train\_model} and \texttt{get\_predictions} methods. Currently, the SBM pipeline has generalized linear models (GLMs), generalized additive models (GAMs), random forests, CatBoost, and FFNNs implemented, but the pipeline is designed to allow for easy integration of custom models in the future. The user can chose one of the current model implementations with specific hyperparameters defined in \texttt{params} block of each model's configuration file, or they can build their own custom model and specify that in the configuration file. The flexibility of the SBM pipeline allows for a wide range of modeling techniques to be used, and promotes the exploration of different approaches to modeling biodiversity patterns and trends at the community level.

\item \textbf{Model training and evaluation}

After the type of training and model has been determined, the model is trained on the dataset. The model is trained to predict the community-level biodiversity metric specified by the user and several metrics will be outputted. For all training types the output will include predictions on both the training and testing datasets, if applicable. The \(R^2\) values, mean absolute error (MAE), mean absolute percentage error (MAPE), symmetric mean absolute percentage error (SMAPE), root mean squared error (RMSE), Pearson correlation coefficient, and Spearman correlation coefficient are also calculated. Additionally, the bias ratios for the train and test datasets are included, along with the bias ratios for the top and bottom quartiles of the train and test datasets. These metrics summarize predictive accuracy, bias, and rank agreement, providing insight into model behavior and suitability for predicting community-level biodiversity patterns. More specific information on how these metrics are calculated will be described in Section~\ref{subsec:model_comparison}.
\end{enumerate}

\subsubsection{Model Application}\label{subsubsec:model_application}
The outputs from the SBM pipeline can then be used in applications downstream. This can be anything from decision making \parencite{IPBES2019spm} to mapping biodiversity patterns which is commonly done in the literature \parencite{andermann2022, baggstrom2025}. The goal is that the SBM pipeline can be used as a tool for researchers to explore the potential of top-down community-level biodiversity models and to expand this research field. The specific application is not defined in the SBM pipeline as this can vary widely depending on the research question and context and is rather left to the user to decide.

\subsection{Application of the SBM Pipeline}\label{subsec:application}
In this subsection we go over the application of the SBM pipeline to a case study using data from GBIF. We use this case study to demonstrate the flexibility and utility of the SBM pipeline in modeling biodiversity at the community level. The details of the data used in this case study are described in Section~\ref{subsubsec:data} and the model selection process is described in Section~\ref{subsubsec:models_spefication}. The top models from each model class are then used to answer the research questions in Section~\ref{sec:Results}.
\subsubsection{Data}\label{subsubsec:data}
%Probably go in depth here a little
For the demonstartion of the SBM pipeline we use the United Kingdom Butterfly Monitoring Scheme (UKBMS) dataset, which is a long-term monitoring program that collects data on butterflies across the United Kingdom and Ireland \parencite{UKBMS2026}. The temporal scale of this dataset is from May 1, 1973 - October 27, 2020, however; when loading the data the date range extends to October 27, 2024. The data is collected from defined transect walks every week from the beginning of April to the end of September, weather permitting to ideally give 26 sample events per site per year. A transect walk is conducted by individuals walking along a defined transect, or route, and counting the number of butterflies they see and recording the species and abundance of butterflies observed \parencite{pollard1993}. At the time of writing this dataset has \(10,209,486\) occurrence records across a few thousand sites. 

We then processed this dataset through the SBM for ingestion and cleaning. An important note here is that we inferred organism quantity as abundance records were separated into individual observations, so we group on latitude and longitude as well as event date. We also checked for coordinate precision down to four decimal places, which is roughly a 10-meter precision. Standard filters of invalid data and invalid dates were also automatically applied. This resulted in \(3,673,775\) abundance records across \(3,038\) unique sites. The spatial distribution of all the sites can be seen in Figure \ref{fig:all_sites_distribution}.

\subsubsection{Model Specification}\label{subsubsec:models_spefication}
%with evaluation of how a model was chosen and the different models that were considered.
% Go more into the details of the statistical methods?? Focus more on top down methods and the ML methods
For the demonstration of the SBM pipeline we chose to train a GLM, GAM, Random Forest, CatBoost, and a FFNN. These models were chosen to represent a range of different modeling approaches, including traditional statistical models (GLM and GAM) and machine learning models (Random Forest, CatBoost, and FFNN) that are commonly used in biodiversity modeling \parencite{ olaya-marin_comparison_2013, mondal2021, andermann2022, brugere2023, cai2023, ngor2023, baggstrom2025}. The GLM and GAM were implemented with \texttt{glm} and \texttt{glmgam} modules respectively from the statsmodels library \parencite{seabold2010statsmodels}. Random Forest was implemented with the \texttt{RandomForestRegressor} class from the scikit-learn library \parencite{scikit-learn}. CatBoost was implemented with the \texttt{CatBoostRegressor} class from the CatBoost library \parencite{catboostdocs2026}. We acknowledge that CatBoost is designed to natively work with categorical features; however, all of our environmental predictors are continuous, and this shouldn't affect how CatBoost is trained. The FFNN was implemented with the \texttt{nn.Module} from pytorch \parencite{pytorchdocs2026}. 

We ran random cross-validation with 10 folds across several different hyper-parameter combinations for each model to evaluate their performance and to choose one model per class to use for our research questions. The best model was chosen based on a combination ranking of the average \(r^2\) score across test folds, the median \(r^2\) across test folds, and the interquartile range of the \(r^2\) across test folds. Every model was ranked according to these metrics, where higher is better for the mean and median \(r^2\) and lower is better for the interquartile range of \(r^2\). The model with the best cumulative rank across these three metrics was chosen as the best model for each class, that is the one with the lowest cumulative rank. These metrics were chosen to give weight to models that on average would perform well while also being robust and having most of the performance be close together. Reasonable values for the hyper-parameter grid search were selected, and the full hyper-parameter search space for each model class is shown in Table~\ref{tab:hyperparameter-grid-searches} in Appendix~\ref{appendix:appendix_b}. The best-performing model from each model class, selected using this evaluation, is shown in Table \ref{tab:best-hyperparameters} below.

We chose to use random cross-validation for model selection for our experiments as it is a good indicator of model performance and is commonly used \parencite{efron_estimating_1983}. We acknowledge that random cross-validation can overestimate model performance when spatial autocorrelation is present in the data \parencite{wang2023}. However, random cross-validation is still a useful tool for model evaluation and is suitable for fairly spread observations \parencite{de_bruin_dealing_2022}. The best-performing model from each model class, was then used in three comparative experiments. First, these models were compared against each other to address Research Question 1. Then, the performance of these models was compared across different cross-validation strategies to address Research Question 2. Finally, the performance of these models was compared across different data scenarios to address Research Question 3. These experiments are described in detail in Section~\ref{sec:Experiments}.

\subsection{Model Comparison}\label{subsec:model_comparison}

The SBM pipeline outputs several evaluation metrics for each model trained, including \(r^2\) values, mean absolute error (MAE), mean absolute percentage error (MAPE), symmetric mean absolute percentage error (SMAPE), root mean squared error (RMSE), pearson correlation coefficient, and Spearman correlation coefficient for both the training and testing datasets. The bias ratios for the train and test datasets are also included, along with the bias ratios for the top and bottom quartiles of the train and test datasets. Evaluating models is holistic process that should not rely on one metric alone.
